# Spec Workflow 使用指南

## 概述

本目录包含项目的规范工作流配置和模板，用于指导项目开发过程中的需求分析、设计、任务分解和实施。遵循这些规范可以提高团队协作效率，确保代码质量和项目进度。

## 目录结构

```
.spec-workflow/
├── README.md                 # 工作流使用指南
├── steering/                 # 项目指导文档
│   ├── product.md            # 产品指导文档
│   ├── tech.md               # 技术指导文档
│   └── structure.md          # 结构指导文档
└── templates/                # 规范模板
    ├── requirements-template.md  # 需求文档模板
    ├── design-template.md    # 设计文档模板
    └── tasks-template.md     # 任务分解模板
```

## 工作流程

### 项目指导文档

项目指导文档位于`steering/`目录，包含三个核心文档：

1. **product.md**: 产品愿景、目标用户、核心功能等产品层面的指导
2. **tech.md**: 技术栈选择、架构设计、数据模型等技术层面的指导
3. **structure.md**: 项目结构、命名规范、代码标准等结构层面的指导

这些文档为整个项目提供统一的指导和标准，应在项目初期制定，并在项目演进过程中保持更新。

### 规范创建流程

对于新功能或Bug修复，请按照以下流程创建规范：

1. **需求分析**：
   - 基于`requirements-template.md`创建需求文档
   - 明确功能需求、非功能需求和验收标准
   - 提交需求文档审核

2. **技术设计**：
   - 基于`design-template.md`创建设计文档
   - 详细设计数据模型、API、UI等
   - 考虑安全性、性能等因素
   - 提交设计文档审核

3. **任务分解**：
   - 基于`tasks-template.md`创建任务分解文档
   - 将功能拆分为可执行的任务
   - 设置任务优先级和依赖关系

4. **实施与验证**：
   - 按照任务列表执行开发
   - 每完成一个任务进行测试验证
   - 记录实施过程和遇到的问题

## 使用方法

### 创建新规范

1. 在项目根目录下创建规范文件夹：
   ```
   mkdir -p specs/[spec-name]
   ```

2. 复制模板文件到规范文件夹：
   ```
   cp .spec-workflow/templates/*.md specs/[spec-name]/
   ```

3. 根据项目需求，修改模板文件内容

### 更新指导文档

当项目需求或技术架构发生重大变化时，应更新`steering/`目录下的指导文档，确保其与项目实际情况保持一致。

## 最佳实践

- **文档先行**：在编码前完成需求分析和设计文档
- **持续更新**：随着项目进展，及时更新文档内容
- **遵循规范**：严格按照文档中定义的标准进行开发
- **定期审核**：定期审核文档和代码，确保质量和一致性
- **团队协作**：鼓励团队成员参与文档评审，集思广益

## 注意事项

- 保持文档简洁明了，避免过于冗长
- 使用具体、可衡量的术语描述需求和验收标准
- 确保文档与实际代码保持同步
- 在文档中记录重要的决策和理由
